# Rule: Hallucination Prevention

## Core Rule
**The SLM MUST only state facts that appear in the retrieved context chunks.**

Fabricating information — even plausible-sounding policies, numbers, or contact details — is a critical failure mode that erodes customer trust and can cause legal/compliance issues.

## Enforcement Mechanisms

### 1. System Prompt Instruction (required)
Every prompt must include:
> "Answer questions using ONLY the information provided in the context below. If the answer is not in the context, say: 'I don't have that information — please contact our support team.'"

### 2. Score Threshold Gate
- Set `score_threshold ≥ 0.6` in `config/rag.yaml`.
- Chunks with low similarity are more likely to introduce off-topic or misleading context.
- Never set threshold below 0.5.

### 3. Fallback Guard (required)
- If `chunks` list is empty, the response MUST be replaced with the fallback message — not generated by the SLM.
- Implemented in `scripts/pipeline/postprocess.py:apply_fallback_guard()`.

### 4. RAGAS Faithfulness Gate
- `faithfulness` score must be ≥ 0.80 before any config or model change ships.
- Faithfulness < 0.80 indicates the SLM is generating content not supported by retrieved chunks.

### 5. Post-process Hallucination Signal Detection
- The post-processor flags phrases like "I believe", "I think", "probably", "might be".
- These signals suggest the SLM is reasoning beyond the context. Log and review.

## Response to Hallucination Detection
1. If RAGAS faithfulness drops below threshold: block deployment, investigate prompt.
2. If tone checker detects hallucination signals: log warning, flag for human review.
3. If customer reports incorrect information: add a corrected Q&A pair to the test dataset.
